Part 7: Multicasting Operators

Performance optimization: share() and shareReplay()

Multicasting Operators

Multicasting operators share observable execution among multiple subscribers, preventing duplicate HTTP calls and improving performance. Critical for production applications.

Key Operators:

  • share(): Hot observable, no replay - Multiple subscribers share execution
  • shareReplay(): Replays last n values to new subscribers - Perfect for caching

⚠️ Common Mistake:

Without sharing, each subscription to an HTTP observable creates a new HTTP request. This can cause performance issues and unnecessary server load.

Key Takeaways

  • ✓ Always use shareReplay() for HTTP observables that multiple components might subscribe to
  • ✓ Use shareReplay({{ '{' }} bufferSize: 1, refCount: false {{ '}' }}) for configuration that should stay cached
  • ✓ Use share() for real-time data where caching isn't needed
  • ✓ Without sharing, each subscription = new HTTP call - major performance issue!
  • ✗ Never expose HTTP observables without sharing - causes duplicate requests
